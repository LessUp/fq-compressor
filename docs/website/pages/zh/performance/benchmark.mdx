# 性能基准测试

## 已跟踪结果集

仓库现在把 benchmark 页面视为“已跟踪结果工件的摘要”，而不是手工维护的宣传表格。

- 跟踪 JSON：`benchmark/results/err091571-local-supported.json`
- 跟踪报告：`benchmark/results/err091571-local-supported.md`
- 数据集清单：`benchmark/datasets.yaml`

## 已验证范围

- 数据集：ENA `ERR091571`
- 实测输入：从 read 1 流式截取的前 2,000 条记录
- 已测本地工具：`fqc`、`gzip`、`xz`、`bzip2`
- 已延后专用对手：`spring`

这是一份“本地支持工具集”结果，不是 FASTQ 压缩器全领域排名。

## 当前结论

对当前跟踪工件而言：

- `fq-compressor` 在压缩比上没有领先已测集合
- `fq-compressor` 在压缩速度上没有领先已测集合
- `fq-compressor` 在解压速度上没有领先已测集合
- 与 FASTQ 专用同类工具的比较仍不完整，因为 `spring` 仍处于延后状态

## 为什么正式工件只用 2,000 条记录

正式 benchmark 必须同时满足“公开来源”和“可重现”。

仓库可以从 ENA 流式准备更大的 20,000-record 子集，但当前 `fq-compressor` 构建在该探索规模上对这一工作负载出现明显性能回归。因此，当前固定工件采用更小但仍然公开可重现的子集。

## 重现方式

```bash
./scripts/benchmark.sh \
  --dataset err091571-local-supported \
  --prepare \
  --build \
  --tools fqc,gzip,xz,bzip2,spring \
  --threads 1 \
  --runs 1
```

## 范围边界

不要把本页理解成普遍排名证明。凡是未实际测量的缺失或延后专用对手，都不在当前“已验证比较范围”内。
