# 参考项目与目录治理说明

## 背景

仓库中曾存在 `ref-projects/` 目录，用于本地保存参考项目源码或镜像副本，例如：

- Spring
- NanoSpring
- minicom
- fqzcomp5
- fastq-tools
- repaq
- pigz
- HARC

这类目录适合调研与对照阅读，但**不适合作为正式项目结构长期保留**，原因如下。

## 为什么不建议继续保留 `ref-projects/`

- **仓库职责不清晰**
  - 正式仓库应该聚焦自身源码、测试、文档和自动化。

- **体积与维护成本高**
  - 参考项目副本通常体积较大，且不受当前项目版本节奏管理。

- **容易造成误解**
  - 新读者可能误以为这些目录是当前项目的 vendor 依赖或必须构建组件。

- **已被 `.gitignore` 忽略**
  - 当前仓库事实上已经不把它当作正式受控内容。

## 推荐替代策略

### 1. 只保留文档结论

将真正需要长期保存的信息，沉淀为：

- 参考项目链接
- 关键算法借鉴点
- 与本项目的差异分析
- 适用场景与局限性

### 2. 本地调研仓库自行维护

如果你仍需要参考源码，建议：

- 在本地单独维护调研目录
- 或使用外部脚本按需 clone
- 或仅在 `vendor/` 中保留真正要集成的代码

## 各参考项目对 fq-compressor 的价值

### Spring / Mincom

- 提供短读场景下 ABC 路线的重要算法启发
- 对本项目的 block-based 随机访问设计提出了适配挑战

### fqzcomp5

- 为质量值压缩提供上下文建模和统计压缩思路

### fastq-tools

- 提供现代 C++ 工程化、脚本入口与流水线设计参考

### repaq

- 提供以压缩比与性能权衡为核心的实现思路

### pigz

- 提供 Producer-Consumer 并行压缩架构参考

### NanoSpring

- 为长读场景提供 overlap-based 思路对照
- 也反向证明：长读与短读不应强行套用同一 ABC 路线

### HARC / minicom

- 提供针对序列压缩与 minimizer / hash 路径的补充视角

## 结论

- **`ref-projects/` 可以从正式项目目录中移除**。
- 建议将其价值收敛为文档，而不是继续作为仓库目录暴露。
- 如果未来确实要纳入某个外部实现，应通过：
  - `vendor/`
  - 明确许可说明
  - 明确构建接入方式

而不是继续沿用 `ref-projects/` 这种模糊定位的目录。
