# GitHub Pages 重构设计稿（fq-compressor）

## 1. 目标与约束

本次重构把 GitHub Pages 从“项目文档集合”提升为 **技术白皮书 / 架构展示站 / 项目学院**。站点需要同时满足三类目标：

1. **第一印象足够强**：让面试官和高级开发者在首页就看到项目的技术主张、证据路径和架构层次。
2. **内容结构足够深**：把“导读 → 原理 → 架构 → 证据 → 实操 → 参考文献”组织成连续学习路径。
3. **工程底座足够现代**：对齐 `/home/shane/dev/kimi-cli` 的 VitePress 技术选型与目录思路，补齐 Mermaid / llms.txt / 更稳的 base 处理，同时修复当前主题在浅深色模式下的可读性短板。

约束：

- 仓库处于 closeout mode，优先“完成、收敛、稳定”，避免引入重型前端框架和难维护动画。
- 文档为双语站点，信息架构必须在中英两侧保持镜像。
- 当前公开站点已能构建，因此重构必须是 **激进但可验证** 的整体替换，而不是无序堆叠。

## 2. 现状审计

### 2.1 当前 fq-compressor 站点

- 已使用 VitePress，并带有自定义首页组件（`MetricStrip`、`KnowledgeMap`、`TopicCardGrid`）。
- 导航结构已经初步按 `overview / whitepaper / architecture / benchmarks / guides / resources` 组织。
- 首页文案已偏技术化，但视觉系统仍偏“单页门户”，缺少更强的论文式秩序、章节引导和图示语言。
- 主题层仅依赖手写 CSS；未引入 Mermaid，也未引入 llms.txt 插件，和 kimi-cli 的底层工程化还未完全对齐。
- 当前 docs 构建正常，但站点没有统一的“图示组件策略”，后续容易继续出现浅/深色适配失衡。

### 2.2 kimi-cli 参考实现

- 依旧使用轻量的 **VitePress + 原生主题扩展 + 定制 CSS**，没有引入额外 UI 框架，维护成本低。
- 在底层工程化上更成熟：  
  - `vitepress-plugin-mermaid`
  - `vitepress-plugin-llms`
  - 更稳健的 `base` 解析逻辑
  - `sync` 型构建前置脚本
- 信息架构更清晰，导航直接面向“使用路径”，而不是面向仓库结构。

## 3. 方案对比

### 方案 A：延续当前门户风格，做局部精修

优点：

- 改动最小，见效快。

缺点：

- 无法彻底完成“白皮书站”转型。
- 继续背负当前首页组件的语义和视觉限制。
- 很难与 kimi-cli 的底层配置完全对齐。

### 方案 B：完全对齐 kimi-cli 底座，重做 fq-compressor 的品牌层与内容层（推荐）

优点：

- 工程底座直接采用更先进的配置组合。
- 保留 VitePress 的轻量与稳定，同时允许 fq-compressor 构建更强的“科研感 / 架构感”视觉。
- 适合 closeout mode：结构升级大，但技术栈简单、长期成本可控。

缺点：

- 需要重写主题样式、首页入口和部分文档目录。
- 需要系统梳理双语内容映射。

### 方案 C：升级为重组件化展示站

优点：

- 视觉表现力最高。

缺点：

- 远超 closeout mode 所需。
- 维护复杂度高，容易把文档站做成前端工程项目。

**结论：采用方案 B。**

## 4. 目标信息架构

新的公开文档树按“学院式阅读链路”组织：

1. **Overview / 导读**
   - 项目定位
   - 为什么值得关注
   - 证据地图
2. **Whitepaper / 白皮书**
   - 问题定义
   - 核心压缩思想
   - ABC / SCM / 重排 / 差分等关键章节
3. **Architecture / 架构**
   - 系统分层
   - 数据流
   - 随机访问与格式
   - 内存与并行执行
4. **Benchmarks / 证据**
   - 基准结论
   - 方法学
   - 局限与公平性说明
5. **Academy / 学院**
   - 安装与快速开始
   - CLI 工作流
   - 面向贡献者的阅读路径
6. **Research / 研究与参考**
   - 参考文献
   - 相关开源项目剖析
   - 演进思考与未来边界

其中第 5、6 两组用于替代当前 `guides` 与 `resources` 的弱语义命名，使站点更接近“技术学院 + 白皮书附录”。

## 5. 视觉系统设计

### 5.1 总体风格

- 以 **学术白皮书 + 工程图册** 为主，不走营销型渐变大屏。
- 视觉气质：冷静、精密、留白充足、代码感强。
- 首页采用“主张—证据—结构—路径—引用”的论文式首屏叙事。

### 5.2 色彩与层次

- 延续蓝色系品牌，但从当前偏“门户卡片”的色彩，升级为更克制的 **研究蓝 / 钢灰 / 数据青**。
- 建立完整 token：
  - 背景层
  - 边框层
  - 高亮层
  - 图表线条层
  - 代码 / 指标层
- 所有 SVG、Mermaid、内联图示统一走 CSS 变量驱动，禁止写死只适合单一主题的颜色。

### 5.3 首页模块

首页重构为：

1. Hero：一句话主张 + 核心指标 + 入口按钮
2. Thesis：项目为何成立、解决什么问题
3. Evidence Grid：压缩率 / 吞吐 / 随机访问 / 并行性四大证据卡
4. Architecture Atlas：从 FASTQ 输入到 FQC 输出的数据流总览
5. Reading Tracks：面试官 / 使用者 / 贡献者 / 研究者四条路径
6. Research Footnotes：论文、竞品、仓库实现线索

## 6. 图示与深浅色模式策略

### 6.1 图示策略

- 引入 Mermaid，用于：
  - 总体架构图
  - 压缩/解压流水线
  - FQC 格式结构
  - 阅读路径与证据链
- 为 Mermaid 设置统一主题变量，保证浅色和深色下均有足够对比度。
- 对需要精致排版的图示，使用 HTML/CSS 组件而不是位图。

### 6.2 深浅色修复

为避免“浅色看不清 / 深色发灰 / SVG 线条消失”：

- 图标与图示全部改为 `currentColor` 或 token 变量驱动。
- Mermaid 主题变量和站点 token 同源。
- 对关键图卡增加显式边框、表面色和文字色，不依赖背景猜测。
- 对首页关键插图做明暗两套阴影与网格强度控制，而不是只切背景色。

## 7. 工程实现蓝图

### 7.1 底层配置

- 对齐 kimi-cli：
  - 使用稳健的 `base` 计算方式
  - 引入 `vitepress-plugin-mermaid`
  - 引入 `vitepress-plugin-llms`
- 保持 VitePress 默认主题扩展模式，不引入额外 UI 库。

### 7.2 主题层

- 简化现有离散组件，构建更统一的 theme primitives：
  - SectionFrame
  - EvidenceCard
  - AtlasFlow
  - ReadingTrackGrid
  - CitationList
- 将样式 token 化，避免内容页和首页各写一套视觉语言。

### 7.3 内容层

- 调整 nav / sidebar，使之围绕“阅读任务”而不是“仓库章节名”。
- 为中英两侧同步新增：
  - Academy/Research 栏目
  - 参考文献页面
  - 相关项目剖析页面
  - 演进思考页面
- 对已有白皮书与架构文档补入：
  - 问题边界
  - 设计权衡
  - 与同类方案对照

## 8. 风险与控制

- **风险：** 一次性改动站点结构较大。  
  **控制：** 保持 VitePress 和现有 Markdown 内容为主体，避免引入复杂构建链。

- **风险：** 双语内容维护量上升。  
  **控制：** 先以结构镜像为第一原则，文案层做高质量等价表达，不做功能分叉。

- **风险：** 图示在主题切换时再次失真。  
  **控制：** 所有图示统一纳入 token 体系，并以构建后站点进行深浅色验证。

## 9. 实施结论

本次重构将采用：

- **技术路线**：VitePress 轻量底座 + kimi-cli 同款工程化插件 + fq-compressor 专属科研风主题
- **内容路线**：从“文档集合”升级为“白皮书 / 架构 / 证据 / 学院 / 研究附录”
- **设计路线**：克制、工程化、学术化、强证据链
