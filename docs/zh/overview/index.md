# 概览

把这一页当作读者导览。它说明 fq-compressor 想证明什么、文档分别服务谁，以及本站剩余部分该怎么读。

## 这个站点的用途

- 把 fq-compressor 呈现为一套 FASTQ 压缩系统设计，而不是一条 benchmark 结果。
- 给面试官和高级开发者一条进入架构、证据与源码锚点的短路径。
- 让操作者和贡献者不必先读完整份白皮书才能开始动作。

## 建议阅读顺序

1. 先读 [白皮书](/zh/whitepaper/) 了解中心命题和主要设计主张。
2. 再读 [架构](/zh/architecture/) 理解模块边界、流水线和格式职责。
3. 再读 [基准测试](/zh/benchmarks/) 确认证据边界与方法。
4. 如果你的问题是操作性的，就进入 [学院](/zh/academy/)。
5. 如果你的问题是对照、历史或参考文献性质的，就进入 [研究](/zh/research/)。

## 每条路径分别服务谁

- **面试官 / 评审者：** 白皮书，然后看基准证据。
- **GitHub 高级开发者：** 架构，然后看研究部分。
- **操作者：** 学院，并在需要时配合架构一起看。
- **贡献者：** 架构、学院，以及研究部分给出的仓库锚点。

## 为什么要这样组织

这个站点的目标，是让公开主张可审计。如果一句关于 fq-compressor 的话，既连不到架构边界，也连不到测量方法或仓库产物，那么它就不应被当成稳定的公开叙事。
