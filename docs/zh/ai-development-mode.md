# AI驱动开发模式分析：简化版

## 概述

本项目曾采用 **Trellis + OpenSpec** 双轨制的AI驱动开发模式。经评估后，针对项目当前阶段（closeout 模式、低频率变更、AI 为主开发者），已简化为**直接对话开发模式**；OpenSpec 仅作为历史资料保留。

---

## 调整历史

### 调整前：Trellis + OpenSpec 双轨制

```
specs/                 # 历史规格资料
├── specs/             # 活跃规格
├── changes/           # 变更提案
└── archive/           # 已完成变更

.trellis/              # Trellis：工作流与知识管理
├── spec/              # 编码规范
├── tasks/             # 任务追踪
├── workspace/         # 会话日志
└── workflow.md        # 工作流定义
```

**问题**：
- 流程过于复杂，对于低频率变更的项目来说是"杀鸡用牛刀"
- 两套系统存在功能重叠（规范、任务管理）
- 学习和维护成本高

### 调整后：直接对话模式

```
项目根目录/
├── CLAUDE.md          # AI 入口文档
├── AGENTS.md          # 完整项目说明（含编码规范）
├── docs/              # 文档
└── .claude/
    └── settings.json  # 基本配置
```

---

## 当前工作流

### 极简版

```
用户请求 → AI 理解需求 → 直接实现 → 运行测试/lint → 提交
```

### 关键原则

1. **直接对话** - 不需要创建任务、不需要子代理，也不依赖 OpenSpec
2. **文档即规范** - AGENTS.md 包含所有编码规范
3. **质量门禁** - 保留 `./scripts/build.sh`、`./scripts/test.sh`、`./scripts/lint.sh`
4. **知识在文档** - 重要决策写入 `docs/` 或代码注释

### 开发检查清单

```bash
# 构建
./scripts/build.sh clang-debug

# 格式检查
./scripts/lint.sh format-check

# 测试
./scripts/test.sh clang-debug
```

## 适用场景

### 直接对话模式适用

- 项目处于维护期或 closeout 阶段
- 变更频率低（每周 < 2 次）
- AI 为主开发者
- 需求清晰，不需要复杂的需求探索

### Trellis + OpenSpec 适用

- 新产品开发，需求模糊
- 大型重构项目
- 需要长期知识积累
- 团队协作，需要规格文件作为沟通基础

---

## 经验总结

### 选择开发模式的关键因素

| 因素 | 简化模式 | 结构化模式 |
|------|----------|------------|
| 变更频率 | 低 | 高 |
| 需求清晰度 | 高 | 低 |
| 项目阶段 | 维护/closeout | 开发/重构 |
| AI 角色 | 主开发者 | 辅助/协作 |
| 知识积累需求 | 低 | 高 |

### 核心权衡

**结构化模式**的价值在于：
- 知识持久化（AI 会话会被压缩）
- 质量可控（子代理分工）
- 可追溯性（完整变更记录）

**简化模式**的价值在于：
- 效率（无流程开销）
- 简单（无学习曲线）
- 灵活（直接对话）

---

## 结论

没有"最优"的开发模式，只有"最合适"的模式。

对于本项目（closeout 模式、低频率变更、AI 为主开发者），**直接对话模式**是更好的选择。

如果项目重新进入活跃开发阶段，可以考虑重新引入结构化工具。
